# 開場

我每次寫技術筆記寫在hackmd就會變雜亂，心想還是好好的把自己學習歷程收納在自己的筆記上，就從慢慢認識Docusaurus開始

使用AI工具時，即使生成程式碼，我是很執意會需要了解三個面向：`do what?` `how?` `why?`，並且針對架構與流程生成Mermaid讓我能夠了解流程的線程，每個節點是否可以替換其他方式？

可能我是學文學的，有點習慣對長文章的`架構` `用途` `寫法`有所掌握才有安全感，這對流程的掌握也是一個提綱契領的好方法！

## 1) 了解Docusaurus根目錄結構

```bash
sui-dev-notes/
├─ .docusaurus/ #=== 開發快取 + 自動產生檔案（不要改）
├─ blog/ # === Blog 文章（時間線/心得/日記）
├─ docs/ # === Docs（技術筆記都寫在這裡）
├─ node_modules/ # npm 套件
├─ src/ #=== 自訂 UI / React 頁面 / 元件
├─ static/ #=== 靜態資源：圖片、favicon（用 /img/... 引用）
├─ .gitignore #===  Git 忽略清單
├─ docusaurus.config.js #===  全站設定：navbar、docs/blog、mermaid、部署等
├─ package-lock.json #===  npm lock（自動生成）
├─ package.json ##===  scripts/依賴（npm run dev 等）
├─ README.md #===  專案說明
└─ sidebars.js #===  Docs 左側 sidebar 設定（目前用自動生成）
```

### Do what 做了什麼？

根目錄，其實Docusaurus 就是分三大區塊：
- 內容（content）：docs/、blog/
- 外觀（UI）：src/
- 資源（assets）：static/
- 設定（config）：docusaurus.config.js、sidebars.js

### How（用我現在的資料夾對照）

> `docs/` 放「文件 / 筆記」分類的地方，用圖書館的館藏說法就是 `技術筆記館藏區XD`

- 這裡應該是我最常駐的地方，所以了解一下，像我現在在HACKMD中大量需要重整的筆記，
我消化完之後，就慢慢移到這裡，最近是用LINE OA讓我的專案做了互動式呈現的方式，打鐵趁熱，剛好整理一下筆記分類+文章架構，也理解這裡資料夾分類的規則：

- **`docs/` 資料夾（category）**  
例如 `LINEOA-PORFOLIO/` 會在 sidebar 出現一個章節

- **文章（doc）**：每個 `.md`/`.mdx` 就是一篇筆記
- **`_category_.json`**：控制這個資料夾（category）分類在 sidebar 的顯示名稱、排序、入口頁

```bash
docs/
├─LINEOA-PORFOLIO/... → 一個技術筆記的分類（章節）
├─ category.json # 這個LINEOA-PORFOLIO資料夾分類的 sidebar 設定
├─ congratulations.md # 範例文章（可改成你自己的）
├─ create-a-blog-post.md # 範例文章
├─ create-a-document.md # 範例文章
├─ create-a-page.md # 範例文章
├─ deploy-your-site.md # 範例文章
├
└─ markdown-features.mdx # MDX 範例（展示 React/MDX 功能）

那我這裡就是專門整理：
我如何建置自己的官方LINE作為呈現互動式專案履歷的技術筆記館藏區
```

- 每個 `.md` 建議都加這段在最上面

```bash

---
title: "這篇筆記的標題" # 文章標題
sidebar_position: 1 #同一個資料夾底下的排序
---

```

--- 

> `_category_.json` →控制該資料夾在 sidebar 的名稱/排序/是否自動產生索引頁

```json
{
  "label": "LINEOA-PORFOLIO", //sidebar顯示的資料夾(分類)名稱
  "position": 1, //sidebar顯示的分類排序（小的排前面）

    "type": "generated-index", //自動生成「分類入口頁」（列出底下所有文章）
    "description": "5 minutes to learn the most important Docusaurus concepts." //入口頁說明文字
  }
}
```
--- 

> `blog/` →放「部落格文章」：比較偏日記、心得、長文、時間線文章，（我有開著 blog 模組，所以會有 `/blog` 路由）

--- 

> `static/` →放「不會被 bundler 編譯的靜態檔」
- 例如：圖片、favicon、大頭照、專案截圖
- 我可以用 `/img/xxx.png` 直接引用。

---

> `docusaurus.config.js` -> 網站大腦（部署/外觀/功能），全站設定：網站標題、navbar、footer、啟用哪些模組（docs/blog）、mermaid、prism 等
- <Highlight>url + baseUrl：GitHub Pages 網址組合（例如 /dev-notes/）</Highlight>
- <Highlight>presets.classic.docs：啟用 docs 模組（讀 docs/）</Highlight>
- <Highlight>presets.classic.blog：啟用 blog 模組（讀 blog/) </Highlight>
- <Highlight>我就啟用了Mermaid，我實在很依賴 => themes + markdown.mermaid：開啟 Mermaid 圖（我可以在 md 直接寫 mermaid）</Highlight>


--- 

> `sidebars.js`
- 控制 docs 左側 sidebar 的生成方式（我目前用 auto-generated）

```js

  tutorialSidebar: [{ type: 'autogenerated', dirName: '.' }],
};

export default sidebars;
```

這裡程式碼的大意是：
- Docusaurus 會掃 docs/ 的資料夾結構
- 自動產生左側導覽列
- 我只要專心「建分類資料夾＋寫 md筆記」，sidebar 就會跟著長

--- 

> `.docusaurus/` →快取 + 自動產生的檔案（我常用`npm run clear` 清它）

報錯常常會指到它，但真正的原因通常在 `docs/` 或 `src/`

---

> `src/` →可以自定義React 元件（例如我寫了一個 Highlight元件）

```bash
src/
├─components/... → 放自己寫的React元件例如 Highlight、Tag、Step
  ├──ui/... → 我統一放ui資料夾（（元件倉庫））
    ├────Highlight.jsx  → <Highlight /> 元件 =>用來「高亮」某段文字
  ├──HomepageFeatures/ → 首頁卡片區塊的 React 元件
  ├──
├─css/
├
├─pages/ 
  ├──index.js → → 首頁（通常 src/pages/index.tsx）
  ├──index.module.css

```

每次寫新的 React 元件（例如 Highlight、Tag、Step），我會統一放ui資料夾

## 2) 給mdx`src/theme/MDXComponents.jsx` 把元件做全域註冊

:::tip 目標
我想在所有`.mdx`筆記裡直接用`<Highlight />`元件，不想每篇都寫import
- 做法是在`src/theme/MDXComponents.jsx`
把 Highlight 註冊成「全站 MDX 都認得的元件」
:::


#### Do what?
- 把我寫好的 Highlight.jsx 變成 全站 MDX 都能用的元件

做到之後：
- 任何 .mdx 文章都可以直接寫 <Highlight>Highlight</Highlight>元件

#### How（怎麼做）?
1️⃣ ) 確認我的 Highlight 元件位置 => `src/components/ui/Highlight.jsx`

2️⃣ ) 建立/新增 `src/theme/MDXComponents.jsx`
```jsx
import React from "react";
import MDXComponents from "@theme-original/MDXComponents";
import Highlight from "@site/src/components/ui/Highlight";

export default {
  ...MDXComponents,
  Highlight,
};

```

3️⃣ ) 重啟 Docusaurus

4️⃣ ) 在任意 .mdx 直接用

```mdx
---
title: "我的筆記"
---

<Highlight color="#f97316">步驟一</Highlight> 先初始化專案

```

#### Why（為什麼可以？）
- <Highlight color="#60a5fa">.mdx</Highlight>本質是「Markdown + React」，所以可以用 

- <Highlight color="#60a5fa">src/theme/MDXComponents</Highlight> 就像「全站 MDX 的公共元件清單」

就是說我把 Highlight元件 放進去，Docusaurus 就會幫我在所有`.mdx`自動準備好這個元件

<Highlight>我只要註冊一次元件，全站都能用。</Highlight> 

:::tip 結論
- 把「內容」跟「UI」分開放，才不會寫筆記寫到一半還要碰 React；
- Docusaurus 的哲學就是：
  - 內容用 md 管
  - 網站用 config 管
  - 外觀再用 src 客製

:::





## 3) `package.json` `scripts`：會用到的指令

npm run dev：本地開發

npm run clear：清快取（遇到怪錯/舊內容不更新就用）

npm run build：打包產出 build/

npm run serve：本地預覽 build 成品

npm run deploy：部署到 GitHub Pages

---

- 把「內容」集中放在 `docs/`：我寫筆記可以很快
- 把「網站外觀」放在 `src/`：要客製首頁/樣式再改這裡(讓我慢慢熟悉React)
- `sidebars.js` 用 `auto-generated`：最適合筆記站 MVP，省掉手動維護 sidebar
- `.docusaurus` 是快取：出現「明明刪了檔案還找得到」這種錯，通常清掉就好
